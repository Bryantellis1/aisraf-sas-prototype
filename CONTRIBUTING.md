# Contributing to AISRAF

AISRAF is a governed package. Contributions are evidence-first and validator-backed. The working model is "no uncontrolled writes": every change traces to a work package, every staging step uses an explicit path list, and every commit follows a green validator ladder.

## 1. No Uncontrolled Writes

- Do not edit canonical surfaces (`prompts/`, `skills/`, `prototype-agents/`, `catalogs/`, `blueprints/`, `templates/`, `config/`, `samples/`, `runs/RUN-001/`) without an approved work package that names the edit.
- Do not edit provider projections (`.agents/`, `.github/agents/`, `.github/skills/`, `.github/hooks/`, `.copilot-skills/`) unless a validator explicitly requires it or an approved work package authorizes it.
- Do not edit the plugin bundle (`plugins/aisraf-copilot-plugin/bundle/`) by hand. Use `tools/Build-AisrafCopilotPluginBundle.ps1 -Clean` to rebuild after changing a canonical or projection file the bundle copies.
- Do not edit the bundle checksum manifest by hand; it is regenerated by the bundle builder.

## 2. No Broad `git add`

- Do not use `git add .`.
- Do not use `git add -A`.
- Do not use `git commit -a`.
- Stage files by explicit path. Use `git add -- <path>` for each file or use a controlled allow-list.
- Inspect `git status --short`, `git diff --name-only`, and `git diff --staged --name-only` before every commit.

## 3. No Generated Smoke Runs In Commits

- `runs/RUN-SMOKE-LOCAL-*/`, `runs/RUN-SMOKE-PLUGIN-*/`, and `runs/RUN-SMOKE-AM3-*/` are local-only smoke evidence folders. They are excluded from staging via local ignore rules and must never be committed or published.
- `Test-AisrafPackage.ps1` Check 14-runs-readme-only emits WARN rows for any smoke run folder present alongside `runs/RUN-001/`. The WARN is intentional; the folders stay local.
- Do not move smoke runs into a different folder to bypass the WARN. If you need long-term smoke evidence, propose a founder-approved smoke-evidence retention policy first.

## 4. Validators Before Commit

Run the full validator ladder before staging:

```powershell
pwsh -NoProfile -File ./tools/Test-AisrafPackage.ps1
pwsh -NoProfile -File ./tools/Test-AisrafBp12AReadiness.ps1
pwsh -NoProfile -File ./tools/Test-AisrafRunProfile.ps1 -RunProfilePath ./runs/RUN-SMOKE-AM3-001/run-profile.yaml -ExecutionReady
pwsh -NoProfile -File ./tools/Test-AisrafRunProfile.ps1 -RunProfilePath ./runs/RUN-SMOKE-PLUGIN-L2B-001/run-profile.yaml -ExecutionReady
pwsh -NoProfile -File ./tools/Test-AisrafRunProfile.ps1 -RunProfilePath ./runs/RUN-001/run-profile.yaml -ExecutionReady
pwsh -NoProfile -File ./tools/Test-AisrafAm3Runtime.ps1 -ContractsOnly
pwsh -NoProfile -File ./tools/Test-AisrafAm3Runtime.ps1 -RunProfilePath ./runs/RUN-SMOKE-AM3-001/run-profile.yaml
```

All validators must return 0 FAIL. Smoke run folders may emit WARN rows (Check 14) — those are expected when classified as local-only, ignored, untracked, unstaged evidence. Anything else FAIL or new WARN must be diagnosed before staging.

If you are working on a controlled-output run, also validate the run profile:

```powershell
pwsh -NoProfile -File ./tools/Test-AisrafRunProfile.ps1 -RunProfilePath ./runs/<your_run_id>/run-profile.yaml -ExecutionReady
```

## 5. Evidence-First Posture

- Every claim in the docs / changelog / release manifest must trace to a validator script under `tools/`, a checklist under `validation/`, or a governed run under `runs/RUN-001/`.
- Every finding produced by the AISRAF review chain must trace to evidence already extracted earlier in the chain. The chain does not invent facts.
- Every recommendation must trace to a finding and to blueprint / question context.
- Every PR / change description must cite the work package it implements (e.g., `WP-12C-REL0-B`).

## 6. Work-Package Discipline

AISRAF changes are organized into work packages (WP-12C-K2, WP-12C-K3, WP-12C-L2B, WP-12C-QA1, WP-12C-ED1, WP-12C-REL0-B, etc.). A work package names:

- The exact files it may create or edit (allow-list).
- The exact files it must not edit (forbidden list).
- The validators it must keep green.
- The work-package status outcomes it may declare.

Contributions that fall outside an approved work package are not accepted. If you have an idea for a change, propose a new work package first.

## 7. No External Execution Claims

Do not introduce code, docs, or release artifacts that claim AISRAF performs:

- Live Jira ticket intake or post-back.
- Live Confluence publication.
- Live Lucidchart adapter use.
- Live MCP runtime.
- Live Anthropic Claude runtime adapter.
- Live Azure AI Foundry, Google ADK, or Microsoft Agent Framework integration.
- Live database-backed runtime.
- Live Terraform or cloud deployment.
- Live cloud runtime, event bus, or telemetry backend.
- Any external post-back execution.

These are deferred AL4 adapter work. They may be discussed only as future / deferred / not implemented.

## 8. No Autonomy Overclaim

Do not contradict the accepted bounded claim that AISRAF v0.1.2 proves AM3 / AL3 local orchestrated multi-agent runtime evidence. When mentioning AM3 / AL3, preserve the limiter: local-only, human-gated, validator-backed, and evidence-bound. The current release does not claim full specialist-generated review output execution, AM4 external tool / post-back execution, AL5 closed-loop autonomy, production operation, external post-back, marketplace publication, or push/publish approval.

## 9. Public-Safety Posture

Do not commit:

- Secrets (API keys, tokens, passwords, connection strings, private certificates).
- Customer PII or production endpoints.
- Internal-only product names or code names.
- Personal paths (e.g., `C:\Users\<name>\...` outside placeholder examples).
- Sales-heavy framing or unsupported readiness claims.

## 10. License And Notice

Until founder / legal selects the final public license, the repository ships under the placeholder `LICENSE` and `NOTICE.md`. Do not contribute code that assumes or grants open-source rights beyond what those files state.
